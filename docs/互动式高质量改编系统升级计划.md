## 1. 背景

当前 `ip-copy-skill` 已经能完成授权校验、结构化蓝图、场景卡、剧本草稿、爆款解说稿骨架、图片/视频 handoff 等流程，但核心文案能力仍然偏“确定性脚手架”。

已确认的问题包括：

- 场景卡多为模板填充，画面感和叙事差异不足。
- 对白生成依赖固定句式，和源剧本角色、语境关系弱。
- 润色更多是规则贴补，不是真正的编剧级打磨。
- story beats 主要按句子截取，不是真正提取冲突、转折、代价、悬念。
- 多集、多幕、多分支结构支持不足。
- 短剧、长剧、电影、剧本杀、互动影游不应共用同一套结构。
- 竖屏短剧默认资产比例应优先 `9:16`，不能默认宽屏。

核心判断：

`ip-copy-skill` 现在适合作为结构控制层，但不能继续让确定性模板假装完成高质量创作。下一阶段应升级为：LLM 负责创作，代码负责约束、校验、流转和额度控制。

## 2. 总目标

把现有 IP-skills 升级为“LLM 驱动 + 结构约束 + 互动确认”的高质量 IP 改编系统，支持：

- 竖屏短剧
- 海外短剧
- 长剧剧本
- 剧本杀
- 互动影游剧本
- 高质量电影剧本
- 后续图片、分镜、视频、音乐技能联动

原则：

- 先读源素材，再判断和改编。
- 默认 `dry_run`，不经用户明确确认不调用 live，不消耗额度。
- LLM 做创作理解、剧情改编、对白、旁白、分镜设计。
- 代码做格式约束、字段校验、时长校验、角色一致性、流程管控。
- 确定性模板只能作为 fallback / scaffold，必须清楚标记，不能冒充最终成稿。

## 3. 目标架构

```text
┌─────────────────────────────────────────────────┐
│  1. 创作层 Creative Layer                        │

│  每个产出环节都必须带格式约束和质量要求            │

│  3. 交互层 Conversation Layer                    │
│  方向确认、改编边界锁定、阶段性确认、单点重生成      │
└─────────────────────────────────────────────────┘
```

## 4. 核心模块

### 4.1 Creative Engine

建议新增目录：

```text
skills/ip-copy-skill/scripts/creative_engine/
- base.py
- offline_engine.py
- live_llm_engine.py
- mock_engine.py
- schemas.py
```

职责：

- 统一创作模型调用入口。
- 默认离线或 mock 模式。
- live 调用必须显式打开，例如 `live_generation=true` 或等效字段。
- 所有 LLM 输出必须走 JSON schema 校验。
- 输出失败时返回错误、缺失字段、重试建议，不静默伪造高质量结果。

### 4.2 FormatAdapter

建议新增目录：

```text
skills/ip-copy-skill/scripts/format_adapters/
- base.py
- vertical_short_drama.py
- overseas_short_drama.py
- long_series.py
- feature_film.py
- murder_mystery.py
- interactive_film_game.py
```

每个 adapter 必须定义：

- 结构层级
- 必填字段
- 节奏规则
- 时长规则
- 质量检查项
- prompt 模板
- 输出 schema
- 下游 image/video/music handoff 需求

格式差异：

| 格式 | 结构单元 | 典型时长 | 核心差异 |
| --- | --- | --- | --- |
| 竖屏短剧 | episode -> beat -> shot | 1-3 分钟/集 | 10-15 秒钩子，强情绪循环，短对白 |
| 海外短剧 | episode -> scene -> beat | 5-10 分钟/集 | 文化适配，对白英文化，节奏略缓 |
| 长剧剧本 | act -> episode -> scene -> beat | 40-45 分钟/集 | 多线叙事，人物弧光，节奏起伏 |
| 电影剧本 | act -> sequence -> scene | 90-120 分钟 | 三幕式，主题弧，视觉叙事 |
| 剧本杀 | phase -> round -> clue | 3-5 小时 | 角色视角分离，信息不对称，推理链 |
| 互动影游 | node -> branch -> consequence | 可变 | 分支树，汇合点，多结局 |

## 5. 标准互动式改编流程

固定流程：

1. 读取源素材。
2. 解析人物、世界观、主线、副线、冲突、反转、风险点。
3. 向用户确认目标格式、成片/成稿长度、改编力度、主视角、风格比例、测试范围、是否 live 生成。
4. 锁定 `creative_brief`。
5. 生成结构大纲。
6. 用户确认结构后，逐 episode / scene / beat 创作。
7. 执行质量检查。
8. 支持单集、单场、单角色对白、单分支重生成。
9. 输出给 `ip-image-skill`、`ip-video-skill`、`ip-music-skill`。

## 9. 测试要求

一期至少增加以下测试：

- 离线模式不消耗额度。
- live 未确认时阻断。
- LLM 输出 schema 校验。
- 多集结构解析。
- 竖屏短剧 adapter 默认 `9:16`。
- 场景卡不能只输出固定模板句式。
- 对白不能只输出固定硬编码句式。
- `script_draft` 示例 task 非空且可运行。
- `build_blueprint` 时长偏差给出明确 warning 或校验结果。
- 图片 handoff 的 `scene` 与 `prompt` 分离。

## 11. 风险控制

- 不要一次性重构所有格式。
- 不要删除现有 fallback，先降级为 scaffold。
- 不要让代码模板继续冒充高质量创作。
- 不要默认 live 调用。
- 不要绕过结构校验直接自由输出剧本。
- 不要未经用户确认擅自改变 IP 设定、角色关系、剧情因果。

## 12. 当前结论

这次升级值得做，但必须分阶段。

当前最优路线：

先把竖屏短剧改编链路做成真正可用的 V1，再扩展海外短剧、长剧、电影、剧本杀和互动影游。

关键工程方向：

`LLM 创作引擎 + FormatAdapter + 多集结构 + dry_run/live 边界 + 质量评估 + 下游 handoff`
