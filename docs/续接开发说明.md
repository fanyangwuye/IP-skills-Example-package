## 音乐 Skill 当前能力

`ip-music-skill` 已支持：

- `build_music_handoff`：从 `polished_script`、`script_draft`、`scene_cards` 或 `blueprint` 生成主题曲和场景 BGM 任务
- `run_music_handoff`：批量执行 `music_handoff.music_tasks` 并生成 `music_generation_manifest.json`
- `generate_music`：文生音乐
- `add_instrumental`：给上传音频配伴奏
- `add_vocals`：给伴奏加人声
- `cover_audio`：上传音频翻唱 / 换风格 / 旋律重制
- `extend_music`：续写已有生成曲
- `upload_extend_audio`：续写用户上传的外部音频
- `replace_section`：替换已有生成曲片段
- `stem_split`：对 PoYo/Suno 生成曲拆分多轨
- `separate_vocals`：对 PoYo/Suno 生成曲做人声/伴奏分离
- `upload_separate_vocals`：对用户上传的外部音频做拆轨/分离

重要实现文件：

- `skills/ip-music-skill/SKILL.md`
- `skills/ip-music-skill/scripts/music_skill.py`
- `skills/ip-music-skill/scripts/poyo_music_client.py`
- `skills/ip-music-skill/scripts/config.py`
- `skills/ip-music-skill/references/workflows.md`

关键规则：

- 离线 `build_music_handoff` 不耗积分。
- live 音乐调用需要 `MUSIC_API_KEY` 或兼容别名 `POYO_API_KEY`。
- 音乐查询端点是 `/api/generate/detail/music`，不要套用图片状态端点。
- `stem_split`、`separate_vocals`、`replace_section` 等已有曲操作需要上游 `task_id` 和 `audio_id`。
- `cover_audio`、`upload_extend_audio`、`upload_separate_vocals` 可传 `audio_path` 或 `audio_url`。
- PoYo 的 `/api/common/upload/stream` 只支持图片/视频，不支持纯 WAV/MP3。
- 如果传本地 WAV/MP3，`poyo_music_client.py` 会自动用 ffmpeg 封装黑屏 MP4 代理文件，上传 MP4，再把返回的 `file_url` 作为音乐接口的 `upload_url`。
- 本地 WAV/MP3 自动代理需要 ffmpeg；可用 `MUSIC_FFMPEG_BIN` / `MUSIC_FFPROBE_BIN` 指定路径。

真实 API 验证：

- 本地 WAV 已验证可自动封装黑屏 MP4 代理并上传 PoYo，返回 `file_url`。
- `cover_audio` 已验证可用 MP4 代理 URL 重构音频并自动下载。
- `generate_music` 已验证可生成古风武侠快节奏纯音乐并自动下载。
- live 音乐模式默认下载所有返回版本，使用 `_variant_02`、`_variant_03` 后缀；可用 `download_all: false` 改回只下载第一版。
- live 音乐模式默认下载封面图，使用 `_cover.jpg` 后缀；可用 `download_cover_images: false` 跳过。
- `run_music_handoff` 可用 `roles` 只跑主题曲或场景 BGM，也可用 `max_tasks` 限制数量，避免一次性消耗太多额度。

当前判断：音乐 Skill 已达到“可收口继续下一个功能”的状态；后续主要是增强，例如更多模型参数、质量评分/筛选、批量拆轨下载。

## 视频 Skill 当前能力与下一阶段工作安排

`ip-video-skill` 阶段 1 已创建。目标不是一开始做完整成片系统，而是先补“从现有文案、图片、音乐资产到视频任务”的确定性骨架。

视频 Skill 的第一原则是 **跨镜头一致性**。不要把一致性当成最后 QA 才检查的附加项，而要从 `video_handoff` 开始就写进每个镜头任务。

必须锁定并逐镜传递：

- 角色长相：脸型、五官、年龄感、发型、发色、体型气质、面部标记
- 服饰：款式、材质、颜色、破损/湿润/血污等状态，是否允许换装
- 道具：武器/手持物的形状、材质、所在手、状态变化
- 场景：空间布局、门窗/桌椅/道路/地标位置、天气与环境物
- 光影色调：主色、光源方向、冷暖、对比度、雾/雨/尘/霓虹等氛围
- 镜头连续性：上一镜结束姿态、本镜起始姿态、本镜结束姿态、下一镜交接
- 多角色关系：轴线、屏幕方向、视线匹配、站位距离，防止越轴和瞬移

这些字段要形成统一的 `continuity_state` / `visual_lock` / `reference_binding`，并被 `build_shot_plan`、`build_i2v_prompts`、`build_t2v_prompts`、`build_edit_decision_list` 共同使用。

参考 `skills-bundle` 的吸收策略：

- `ai-storyboard-protocol`：作为统一分镜卡和多角色轴线/视线协议的核心来源
- `seedance-action-director`：吸收动作戏、武侠、高速/超高速、打击感与安全尾巴
- `actor-audition-prompt`：吸收表演、潜台词、微表情、对白/字幕/声音节奏
- `trailer-blueprint-director`：吸收信息单元、锚点法、逐镜剪辑表、音乐音效和 EDL 思路
- `ip-adaptation-director`：只吸收交付形态路由思想；不要与现有 `ip-copy-skill` 重复

### 阶段 1：Video Handoff 结构层

已新增 `skills/ip-video-skill`，当前已实现离线不耗积分能力：

- `build_continuity_bible`：从角色图、角色资产包、场景图、剧本状态生成全局一致性圣经
- `build_shot_plan`：把每个 scene card / blueprint segment 转成镜头计划
- `build_video_handoff`：从 `polished_script`、`blueprint`、`image_handoff`、`ip_asset_pack`、`music_handoff` 生成视频任务清单
- `build_i2v_prompts`：根据角色图、场景图、镜头运动生成图生视频提示词
- `build_t2v_prompts`：没有参考图时生成文生视频提示词
- `build_edit_decision_list`：输出初版 EDL，包括镜头顺序、时长、字幕、旁白、BGM 引用

阶段 1 只写 JSON handoff 和 prompt，不调用外部视频 API。每个镜头都会带 `visual_lock`、`reference_binding`、`continuity_state`；多角色镜头会带 `axis`、`screen_direction`、`eyeline`、`blocking_distance`。

当前提示词质量层已增强：

- `prompt_profile`：每镜拆出叙事目标、动作流程、表演控制、镜头控制、空间连续性、光线质感、声音、真实感锚点、执行边界、重抽建议
- `i2v_prompt`：面向图生视频，强调参考图锁脸、锁衣、锁景、锁道具、锁风格，防止参考污染
- `t2v_prompt`：面向文生视频，显式写入角色锁、场景锁和连续性圣经
- `seedance_prompt`：面向 Seedance / 即梦类中文视频提示词，包含表演、镜头、空间、光线、声音和执行边界
- `negative_prompt`：包含脸漂移、发型漂移、服装变色、场景重置、越轴、视线错位、塑料皮肤、过度磨皮等常见失败约束
- `retry_advice`：按失败类型给出重抽建议，例如脸漂移、服装漂移、场景重置、越轴、视线错位、镜头过晃

视频 provider 接入层已开始设计并落地 dry-run 请求层：

- `provider` 的意思是具体视频平台适配器，例如即梦 CLI、Seedance、可灵、PoYo video
- 当前新增 `prepare_video_generation`：从 `video_handoff.shots` 选一个镜头，生成 provider 请求 JSON，不调用外部 API
- 当前新增 `run_video_generation`：默认 dry-run，只写请求/结果 manifest，不真实生成视频
- 当前支持请求形状：
  - `offline` / `dry_run`：只准备请求，完全离线
  - `dreamina_cli`：生成官方 `dreamina` CLI 形状的请求，执行前必须先跑 `dreamina <subcommand> -h` 确认参数
  - `jimeng_cli`：兼容别名，实际入口仍按官方 `dreamina`
  - `poyo_video`：已可按 PoYo Seedance 2 文档提交 `/api/generate/submit`，轮询 `/api/generate/status/{task_id}`，并下载 `files[].file_url`
- provider 请求会保留 `prompt`、`negative_prompt`、`reference_images`、`reference_binding`、`continuity_state`、`visual_lock`、`axis`、`screen_direction`、`eyeline`、`retry_advice`
- PoYo video live 仍默认 dry-run；只有任务显式 `dry_run: false` 且配置 `VIDEO_API_KEY` 或 `POYO_API_KEY` 才会真实提交，优先只做单镜头小样，不批量生成整集

### 阶段 2：视频模型适配层

在结构层稳定后，再接 provider：

- Seedance / 即梦 / 可灵 / PoYo video 如果有 API，按 provider adapter 接入
- 支持 `image_to_video`、`text_to_video`、`extend_video`、`video_reference_motion` 等模式
- 所有 live 任务保持 `run_task(task_dict)` 入口
- 不硬编码 key，新增 `VIDEO_PROVIDER`、`VIDEO_API_KEY`、`VIDEO_API_BASE`、`VIDEO_OUTPUT_ROOT`
- 支持只跑单镜头小样，避免一次生成整集浪费额度

### 阶段 3：成片组装层

先用本地 ffmpeg 做最小可用成片：

- 下载/收集生成视频片段
- 按 EDL 拼接视频
- 叠加 BGM / 音效 / 旁白音轨
- 生成字幕文件 `.srt`
- 可选烧录字幕
- 输出最终 `mp4`

阶段 3 可以先支持“静态图片 + BGM + 字幕”的 fallback 成片，等视频生成接口稳定后再切换为动态片段。

### 阶段 4：质量控制与回环

- 检查镜头数量、时长是否匹配 blueprint
- 检查角色一致性引用是否丢失
- 检查字幕是否超长、是否遮挡画面
- 检查 BGM 时长是否够用
- 输出需要补图、补视频、补音乐的缺口列表

5. 再验证一次 `stem_split` 或 `separate_vocals`
