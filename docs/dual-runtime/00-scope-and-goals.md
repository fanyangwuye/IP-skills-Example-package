# 00 Scope And Goals

## 1. 背景

`ip-skills_CZ` 当前主要面向本地代码型 agent 运行：

- agent 读取仓库内的 `SKILL.md`
- agent 调用 `skills/*/scripts/*.py`
- 输入和输出大量依赖本地文件路径与 `outputs/`
- provider live execution 默认受控，离线 dry-run 和本地校验能力较强

这套模式适合 `Codex`、`Claude Code`、`OpenClaw` 一类可读仓库、可运行 Python、可管理本地文件系统的 agent，但不直接适合纯云端托管型平台 agent。

## 2. 问题定义

当前仓库缺少以下能力：

- 缺少平台无关的服务层入口，云端 agent 无法稳定调用
- 缺少统一的 task 与 manifest 契约文档，平台接入方容易“猜接口”
- 缺少云端文件模型，现有流程默认把本地路径作为一等公民
- 缺少异步任务与产物回传模型，长链路工作流难以稳定编排
- 缺少 `Coze` 这类平台的工具拆分与工作流接入方案

## 3. 目标

本次双适配设计的目标是：

1. 保留当前本地 agent 工作流，不破坏现有 skill 入口与离线测试能力
2. 提炼一套平台无关的核心 task/manifest 合同，供本地和云端共用
3. 在核心逻辑之外新增云端适配层，而不是复制业务脚本
4. 让云端 agent 能通过 API/工具/工作流稳定调用 `ip-copy-skill`、`ip-video-skill` 等核心能力
5. 保持 provider 执行边界清晰：核心层负责准备、校验、审计，不默认承担任意 live 调用

## 4. 非目标

本阶段不做以下事情：

1. 不把整套仓库改写为仅支持云端的 SaaS 项目
2. 不把所有 skill 一次性改造成完整 GUI 产品
3. 不承诺立即支持所有平台；先以“平台无关架构 + Coze 优先适配”作为目标
4. 不在没有外部执行层设计的情况下，把 live provider 调用直接塞进平台节点
5. 不以牺牲 deterministic 校验和质量门禁为代价换取快速平台接入

## 5. 目标运行时

### 5.1 本地运行时

支持以下特征：

- 可读取仓库文件
- 可运行 Python
- 可访问本地文件系统
- 可使用环境变量或本地密钥配置

典型对象：

- `Codex`
- `Claude Code`
- `OpenClaw`
- 其他具备代码执行与文件访问能力的 agent

### 5.2 云端运行时

支持以下特征：

- 通过 API、工具调用或工作流节点调用能力
- 默认不能直接读取仓库目录结构
- 默认不能依赖本地路径作为长期协议
- 需要显式的文件上传、URL、对象存储或文件 ID 机制

典型对象：

- `Coze`
- 其他托管型云端 agent 平台

## 6. 核心设计原则

1. 一套核心逻辑，多套适配层
2. 先定义合同，再定义 API，再定义平台节点
3. 本地路径不能成为唯一协议
4. provider live execution 与业务编排解耦
5. 流程约束、quality gate、审计信息必须在双运行时保持一致

## 7. 成功标准

双适配方案至少应满足以下成功标准：

1. 同一个核心 task 在本地和云端能产生等价的结构化结果
2. 本地模式仍可通过现有离线测试，不因云端适配而破坏
3. 云端模式可通过稳定 API 或工具调用 `ip-copy-skill` 和 `ip-video-skill` 的核心能力
4. 输入、输出、错误、产物引用、任务状态都有明确定义
5. 平台接入方无需阅读全部仓库源码就能正确调用核心能力

## 8. 阶段划分

### Phase 1: 设计定稿

- 完成双适配架构文档
- 完成核心 task/manifest 契约文档
- 完成存储、执行、provider 边界文档

### Phase 2: 核心层抽象

- 从现有脚本中提炼稳定核心 entrypoint
- 统一输入输出 schema
- 抽象文件引用与产物模型

### Phase 3: 本地适配收口

- 保持现有 `SKILL.md + Python` 模式
- 把现有本地脚本接到统一核心层
- 保持离线测试通过

### Phase 4: 云端适配落地

- 实现 API/service layer
- 实现异步 job 模型
- 实现平台文件传递与产物回传

### Phase 5: 平台适配

- 优先完成 `Coze` 工具/API/工作流接入
- 补平台级集成测试与示例

## 9. 首批优先范围

首批双适配优先覆盖两块能力：

1. `ip-copy-skill`
   原因：它承担 source -> scene/script/asset pack 的前置控制层，适合作为云端调用入口
2. `ip-video-skill`
   原因：它承担 continuity、shot/clip、prompt、preflight、provider request 的后置控制层，最能体现双适配设计价值

`ip-image-skill` 与 `ip-music-skill` 在首批中保持兼容，但不作为第一阶段的主拆解对象。

## 10. 交付物

本设计阶段需要输出：

- 双适配文档清单与依赖关系
- 架构边界定义
- task 与 manifest 契约
- 存储与产物模型
- 本地和云端适配设计
- Coze 平台接入方案
- 测试与迁移计划

## 11. 风险

1. 如果不先定义边界，后续容易演变为两套逻辑并行维护
2. 如果不先统一 task/manifest，平台接入会长期依赖隐式约定
3. 如果不抽象文件模型，云端接入会被本地路径强耦合卡死
4. 如果把 provider live execution 直接放进平台层，审批、审计和失败追踪会失控

## 12. 本文档结论

双适配方案是可行的，但必须先把仓库从“本地 agent 优先的脚本仓库”提升为“核心能力稳定、适配层可替换的能力仓库”。后续所有接口和平台设计，都必须以这个目标为前提。
